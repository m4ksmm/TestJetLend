# TestJetLend

Основная задача — реализовать endpoint `POST /api/orders/`, где пользователь передает список товаров и опциональный промокод. Промокод дает процентную скидку, если проходит проверки по сроку действия, лимиту использований, пользователю, категориям и исключенным из акций товарам.

## Стек

- Python 3.10+
- Django 4.2+
- Django REST Framework
- PostgreSQL
- Docker Compose

## Запуск локально

Сначала нужно установить зависимости:

```bash
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

Поднять PostgreSQL:

```bash
docker compose up -d postgres
```

Применить миграции:

```bash
python manage.py migrate
```

Создать демо-данные:

```bash
python manage.py seed_demo_data
```

Запустить сервер:

```bash
python manage.py runserver
```

По умолчанию проект подключается к базе:

```text
postgres://jetlend:jetlend@localhost:5432/jetlend
```

## Тесты

```bash
pytest
```

Тесты используют PostgreSQL. Это важно, потому что для защиты лимита промокодов используется `select_for_update()`, а такая блокировка нормально проверяется именно на базе с row-level locking.

## API

### Создать заказ

```http
POST /api/orders/
```

Пример запроса:

```bash
curl -X POST http://127.0.0.1:8000/api/orders/ \
  -H "Content-Type: application/json" \
  -d '{
    "user_id": 1,
    "goods": [
      {
        "good_id": 1,
        "quantity": 2
      }
    ],
    "promo_code": "SUMMER2025"
  }'
```

Пример ответа:

```json
{
  "user_id": 1,
  "order_id": 1,
  "goods": [
    {
      "good_id": 1,
      "quantity": 2,
      "price": "100.00",
      "discount": "0.1000",
      "total": "180.00"
    }
  ],
  "price": "200.00",
  "discount": "0.1000",
  "total": "180.00"
}
```

`price` у позиции заказа — это цена одной единицы товара на момент создания заказа.  
`total` у позиции — итог по позиции с учетом количества и скидки.

## Демо-данные

Команда:

```bash
python manage.py seed_demo_data
```

создает:

- пользователей `demo` и `second_user`;
- категории `Books`, `Electronics`, `Gift Cards`;
- несколько товаров;
- промокоды `SUMMER2025`, `BOOKS10`, `LIMIT1`, `EXPIRED`, `INACTIVE`.

После запуска команда выводит реальные ID пользователей и товаров, чтобы их можно было сразу использовать в curl-запросах.

## Архитектура

Проект разделен на несколько слоев:

```text
config/
    settings.py
    urls.py

apps/
    shop/
        models.py
        admin.py
        api/
            serializers.py
            views.py
            urls.py
        services/
            order_service.py
            pricing.py
            promo_service.py
        management/
            commands/
                seed_demo_data.py
        tests/
```

`api` отвечает только за HTTP-слой: принять запрос, проверить базовый формат данных, вызвать сервис и вернуть ответ.

`services` содержит бизнес-логику. Создание заказа находится в `OrderService`, проверки и расчет применимости промокода — в `PromoCodeService`.

`models` описывает структуру данных и ограничения на уровне базы.

## Модели

В проекте используются модели:

- `Category`;
- `Product`;
- `PromoCode`;
- `Order`;
- `OrderItem`;
- `PromoCodeUsage`;
- стандартный Django `User`.

Цены и итоги хранятся через `DecimalField`. В заказе и позициях заказа сохраняются snapshot-значения: цена, скидка и итог фиксируются на момент создания заказа.

## Логика промокодов

Промокод применяется, если:

- он существует;
- он активен;
- срок действия не истек;
- лимит использований не превышен;
- пользователь еще не использовал этот промокод;
- товар подходит по категории промокода;
- товар не помечен как исключенный из акций.

Если промокод указан, но часть товаров не подходит под условия, заказ все равно создается. Скидка применяется только к тем позициям, которые подходят.

Создание заказа выполняется внутри `transaction.atomic()`. Промокод загружается через `select_for_update()`, чтобы два параллельных запроса не смогли одновременно использовать последний доступный лимит.

Дополнительно в базе есть уникальное ограничение на пару `user + promo_code` в `PromoCodeUsage`.

## Обработка ошибок

Ошибки возвращаются в JSON-формате.

Пример:

```json
{
  "promo_code": ["Promo code has expired."]
}
```

Или:

```json
{
  "user_id": ["User does not exist."]
}
```

## Что покрыто тестами


- создание заказа без промокода;
- создание заказа с валидным промокодом;
- несуществующий промокод;
- просроченный промокод;
- превышение лимита использований;
- повторное использование промокода одним пользователем;
- ограничение промокода по категории;
- исключенные из акций товары;
- неверное количество товара;
- несуществующий товар;
- несуществующий пользователь;
- пустой список товаров;
- смешанный заказ, где часть товаров со скидкой, часть без;
- конкурентное использование промокода с лимитом `max_uses=1`.
